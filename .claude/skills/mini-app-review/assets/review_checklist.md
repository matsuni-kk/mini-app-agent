# レビューチェックリスト

## 1. 機能レビュー観点

### 要件充足
- [ ] 全Must機能が実装されている
- [ ] 全Should機能が実装されている（または明示的にスコープ外）
- [ ] 受け入れ基準を満たしている
- [ ] 当初の目的・ゴールを達成している

### ユーザビリティ
- [ ] 操作が直感的である
- [ ] 適切なフィードバックがある（ローディング、成功、エラー）
- [ ] 誤操作を防ぐ設計になっている
- [ ] 元に戻せる操作がある（必要な場合）

### エッジケース
- [ ] 連続クリック・連打への対応
- [ ] 空データ・未入力時の動作
- [ ] ネットワークエラー時の動作（外部API使用時）
- [ ] 想定外の入力への対応

## 2. コードレビュー観点

### 構造・設計
- [ ] ファイル構成が適切
- [ ] 関数が適切に分割されている
- [ ] 単一責任原則を守っている
- [ ] DRY原則を守っている（重複コードがない）

### 可読性
- [ ] 変数・関数名が意味を表している
- [ ] コメントが適切（過不足なし）
- [ ] 一貫した命名規則
- [ ] 適切なインデント・フォーマット

### 保守性
- [ ] マジックナンバーがない（定数化されている）
- [ ] ハードコーディングが最小限
- [ ] 設定値が分離されている
- [ ] 将来の変更に対応しやすい

### パフォーマンス
- [ ] 不要なDOM操作がない
- [ ] イベントリスナーが適切に管理されている
- [ ] メモリリークの可能性がない
- [ ] 重い処理がブロッキングしていない

### セキュリティ
- [ ] XSS対策（innerHTML使用時の注意）
- [ ] 入力値の検証
- [ ] 外部リソースの安全性
- [ ] 機密情報のハードコーディングがない

## 3. デザインレビュー観点

### UI
- [ ] 視覚的一貫性がある
- [ ] 適切な余白・配置
- [ ] 配色が適切（コントラスト、ブランド）
- [ ] タイポグラフィが読みやすい

### UX
- [ ] 操作フローが自然
- [ ] 重要な要素が目立つ
- [ ] 状態変化が分かりやすい
- [ ] アニメーションが適切（過度でない）

### レスポンシブ
- [ ] モバイルで使いやすい
- [ ] タブレットで使いやすい
- [ ] デスクトップで使いやすい
- [ ] タッチ操作に対応

### アクセシビリティ
- [ ] キーボード操作可能
- [ ] フォーカスが見える
- [ ] 適切なARIA属性
- [ ] 十分なコントラスト比
- [ ] 代替テキストがある（画像使用時）

## 4. 技術的負債チェック

### リファクタリング候補
- [ ] 長すぎる関数がないか
- [ ] 深いネストがないか
- [ ] 複雑な条件分岐がないか
- [ ] 似たようなコードの重複がないか

### 改善余地
- [ ] より良いAPIやメソッドがないか
- [ ] パフォーマンス改善の余地がないか
- [ ] モダンな書き方に置き換えられないか
- [ ] ライブラリで簡略化できないか

## 5. 拡張性チェック

### 将来の機能追加
- [ ] 新機能を追加しやすい構造か
- [ ] 設定やデータの追加が容易か
- [ ] コンポーネントの再利用が可能か
- [ ] テストが追加しやすいか

## 評価基準

| 評価 | 基準 |
|------|------|
| Good | 問題なし、ベストプラクティスに従っている |
| Fair | 軽微な問題あり、改善の余地あり |
| Poor | 重大な問題あり、早急な対応が必要 |

## 優先度基準

| 優先度 | 基準 |
|--------|------|
| High | ユーザー体験に直接影響、バグ、セキュリティ問題 |
| Medium | コード品質、パフォーマンス、保守性に影響 |
| Low | 将来的な改善、リファクタリング候補 |
